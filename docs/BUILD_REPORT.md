# 빌드 회고 로그 — 중규모 코퍼스 v1

각 단계 종료 시 회고를 기록한다. 평가 기준은 항상 스펙 2장(목표)과의 정합성.

---

## 회고 1 — 개념 인벤토리 v0 (1차 슬라이스)

### 잘된 것
- 충돌 패밀리 6개(등급·상태·상환방식·한도·연체·잔액)가 모두 "낱개 distractor"가 아니라
  도메인 간 자연 충돌로 설계됨 — 스펙 8장 의도 충족.
- 기존 자료의 잠재 모순('대출상태'가 신청상태/계좌상태로 혼용)을 발견,
  명시적 입도 함정으로 전환. 버그를 케이스로 승격한 사례.

### 결정 확정 (자율 결정 — 사용자 위임에 따름, 전부 재검토 가능 플래그)
- **D6. `expressed_as` 역할 추가 승인.** 근거: "기한이익상실 건 총액", "리볼빙 비중",
  "VIP 고객의 ~" 등 컬럼+값 술어로 실현되는 개념이 별도 질문 패턴을 라우팅함.
  스펙 7장 기준("질문 패턴과 짝") 통과. → 역할 어휘 7개. 스펙 v0.1 반영.
  판별 테스트: "이 개념은 컬럼인가, 컬럼의 특정 값인가."
- **D7. 신청상태/계좌상태 분리 확정.** '대출상태'는 양쪽에 걸치는 의도적 모호 동의어.
- **D8. 도메인 미지정 모호 개념(해지·연체·잔액·골드등급)의 골든 라벨 정책:**
  (a) 확인 요청 = 정답, (b) 가정을 명시하고 한쪽 답변 = 부분 정답(가정 명시 필수),
  (c) 무가정 단일 답변 = 오답. 평가 하니스에 3단 채점으로 반영.
- **D9. 메트릭 기준 필터 함정 유지** (연체율 상각·완제 제외, 승인율 분모, 동의율 X 제외).
  get_metric 사용 vs 소박한 재계산을 가르는 핵심 장치.

### 발견된 리스크 → 다음 단계
- 역할 어휘 7개가 실제 질문에서 작동하는지 양산 전 검증 필요 → **2단계: 질문 드라이런**.
- 양산 시 규모 목표(컬럼 600~1500, Term 200~500)가 테이블 37개 골격과 정합한지 재계산 필요.

## 회고 2 — 질문 드라이런 (양산 전 검증)

### 결과
- 라우팅 통과 11/15. 역할 7개 전부 슬라이스 안에서 실현 확인 (expressed_as 7회 — D6 타당성 보강).

### 발견 (양산에 반영)
- **F-1. 차원의 코드체계 의무화**: segmented_by/attribute_of로 실현되는 코드성 컬럼
  (REGION_CD, BIZ_TYPE_CD, TAX_EXMP_RSN_CD)은 코드체계 없이는 값 수준 질문이 성립 불가.
  → 양산 규칙: 차원 역할 컬럼은 코드체계 필수 (의도된 경계 결손 제외).
- **F-2. 시간 기준 규약**: "지난달" 류 상대시간은 골든 답의 결정성을 깨뜨림.
  → v1 규약: CORPUS_AS_OF=2026-06-15 고정, 질문 텍스트는 명시적 기간 사용.
  상대시간 해석은 별도 역량 축으로 v2 이연. (스펙 13장 열린 항목에 추가)
- **F-3. '만기'의 잠재 모호성**: DEPOSIT에 만기일(MTRT_DT)이 생기면 '만기'가
  대출/예금에 걸침 → 의도적 모호 동의어로 승격 (충돌 패밀리 F7_maturity).

### 규모 목표 조정 (스펙 8장 대비)
- 테이블 37 (목표 40~60 하한 근접 — 유지), 컬럼 예상 ~450 (목표 600~1500 미달 — **조정**:
  목표치는 테이블 60개 전제였음. 동명 산재·배경 컬럼 밀도는 충분. 부족분은 2차 도메인 확장으로),
- Term 예상 ~130 (목표 200~500 미달 — **조정**: 기능(질문 라우팅 또는 distractor) 없는
  패딩 Term은 평가 노이즈만 추가. distractor 20개 포함 ~130으로 v1 확정, 2차 확장).

## 회고 3 — 양산·레이어·DB·질문셋·검증 (3~7단계)

### 결과 요약
- 인벤토리 양산: 테이블 37 · 컬럼 296 · 코드체계 29(미등재 2) · 개념 81 + distractor 21 = Term 100 · 메트릭 11
- 골든 레이어: terms/columns/tables/metrics/codedict/meta 6파일 + 평가 전용 ground_truth(레이어 비공개)
- mock DB: 37테이블 ~25.7만 행, seed=42 결정적, 정합성 규약 5종 내장
- 질문셋: 44문항 (normal 25 / family 8 / granularity 4 / boundary 4 / join 3), 골든 답 전부 실행 산출
- 검증 V1~V5 전체 통과. 함정 작동 확인: 연체율 0.0853 vs 0.0763, 승인율 0.7149 vs 0.6272,
  동의율 0.4037 vs 0.2998 — 정본 메트릭을 안 쓰면 틀리는 구조가 숫자로 성립.

### 잘된 것
- "정답 먼저, 파생 나중" 원칙이 끝까지 유지됨 — 모든 질문의 골든 답이 생성 시점에 기계적으로 나옴.
- 경계 결손이 레이어에 새지 않음을 기계 검증(V4) — Description 증강 시 코드값이 누출되는
  실수를 검증기가 잡도록 설계해, 사람 눈에 의존하지 않음.
- 중단 사고(빌드 스크립트 truncation) 후 모놀리식 → 단계별 스크립트로 전환, 회복 비용 감소.

### 정직한 한계 (스펙 13장에 등재)
- **L1.** 컬럼 296 — 스펙 목표(600~1500) 하회. 동명 산재(CUST_NO 14개 테이블 등)와 충돌 구조는
  충분하나, '검색 천장' 검증에는 부족할 수 있음. NL 프로토 결과를 보고 2차 확장 판단.
- **L2.** Term 100 — 동일 사유로 목표 하회. distractor 21개 포함, 기능 없는 패딩은 배제 원칙 유지.
- **L3.** 상환 스케줄 회차 ≤12 단순화, 잔액·이자 산술이 약식 — 시계열 정밀 질문(이자 검산류)은
  v1 질문셋에서 의도적으로 배제했고, 그 범위 안에서만 골든이 유효.
- **L4.** 질문 44문항은 통계적 평가엔 작음 — 카테고리별 신뢰구간을 논하려면 80~120문항 필요.
  생성기가 있으므로 증분 비용은 낮음(질문 추가는 스펙 작성 + 실행).
- **L5.** clarify 채점(D8 3단)은 정책만 정의 — 실제 채점기는 NL 프로토 단계의 평가 하니스 몫.

### 다음 단계 입력 (NL 에이전트 프로토타입)
- 레이어 읽기 인터페이스 8연산을 layer/*.json 위에 구현 (스펙 5장 시그니처 그대로)
- 에이전트 루프: 질문 → 연산 호출 → SQL → world.db 실행 → 답
- 평가 하니스: questions.json 골든 대비 채점 (sql 비교가 아니라 실행 결과 비교 + D8 정책)

## 회고 4 — NL 에이전트 구현 (8~10단계)

### 구현 방식 결정 (사용자 지시)
- Python 하니스 단독이 아니라 **기존 Render/Link PoC 양식의 웹앱**으로 — 이식 전제.
- 단, PoC와 달리 채점 하니스가 본체이므로: **순수 로직(8연산·채점기·루프 코어)을 UI에서 분리**해
  node 헤드리스로 먼저 검증 후 UI를 입히는 순서를 지킴.

### 검증 결과 (모델 투입 전)
- 하니스 자가검증: **oracle 44/44 correct** + 결함 에이전트 8종 전부 검출
  (메트릭 함정 무시→오답, 코드 추측→환각, 라벨 발명→환각, D8 무가정→오답/가정명시→부분,
   임의 기준 발명→오답+환각, tool-miss 표지).
- 루프 코어: 정상 경로(연산3→SQL→채점 correct), 중복 호출 가드, 상한 강제 종결 — 통과.
- 8연산 스모크: '연체율'→대출/카드 양 후보 표면화, 기한이익상실→'03' 변환,
  CHNL_CD→명시적 사전 부재, CARD↔DEP 다홉 경로, 수수료 폴백 — 전부 의도대로.
- JSX 변환·전역 적재 스모크 통과. (브라우저 실런은 모델 키 필요 — 사용자 환경에서)

### 설계 결정 (추가)
- **D10. 중립 강화**: resolve_terms는 내부 랭킹으로 top-k 선별 후 이름순 정렬해 순위 신호 제거,
  Term의 family(충돌 패밀리 소속)는 절대 비노출 — '함정 힌트' 누출 차단.
- **D11. tool-miss 면책 표지**: 검색 연산 0건 반환을 플래그로 분리 — 도구 실패가
  에이전트 점수를 오염시키지 않도록 채점에서 식별 가능.
- **D12. 환각의 구조적 검출**: 결손 컬럼 리터럴 필터·CASE 라벨 매핑을 정규식으로 검출 —
  LLM 심판 없이 기계 채점. assumptions 필드 구조화로 D8도 기계 채점.

### 남은 것 (사용자 환경에서)
- API 키 주입 후 '전체 실행' — 첫 실측 점수. 결과 JSONL 다운로드로 오답 분석.
- 실측에서 tool-miss가 잦으면: 검색(도구) 개선이지 에이전트 문제가 아님 — L1/L2 확장 판단 입력.

## 회고 5 — 1차 실측 분석과 보정 (모델: 사용자 로컬 실행)

### 1차 실측 원점수: 30/44 (68%) · 경계 4/4 · 환각 0
**그러나 오답 13건 전수 해부 결과, 측정된 것의 대부분은 에이전트가 아니라 하니스·코퍼스였다.**

### 3버킷 분해
- **A. 채점 엄격성 (8건)** — 값-동치인데 오답: 백분율 표현(0.1249 vs 12.49),
  무반올림(상대오차 6e-5), 정당하게 획득한 추가 컬럼(고객명·valid_values 라벨·보조 카운트),
  피벗 형태. → **채점기 v2**: 상대오차 2e-3, 비율형 한정 ×100 동치, 골든컬럼⊆에이전트
  부분집합 매칭. v2 재채점 시 8건 전부 correct, 함정 비혼동 확인(소박-백분율은 여전히 wrong).
  oracle 100% + 결함 검출 유지. **보정 점수 38/44 (86%)**.
- **B. 코퍼스 저작 버그 (4건, 내 버그)** — ①고아 메트릭: M_APRV_RATE 등 5개가 metrics.json에만
  있고 어떤 Term도 measured_by로 연결 안 함 → 에이전트가 도달할 경로 부재. N12에서 에이전트가
  "정본 부재, 분모 전체 가정"을 명시한 것은 모범 행동이며, 거버닝 메트릭 부재 시 답이 흔들린다는
  CH4 '재유도 가능' 계층의 우발적 실증. ②N25 '연체' — F5 설계상 모호한데 단일 SQL 기대(에이전트
  clarify가 방어 가능). ③G03 '대출상태' 동의어를 계좌상태에만 등재 — 의도한 모호성 미구현.
  → 패치: 메트릭-Term 연결 5건, '대출상태' 동의어 양측 등재, N25 도메인 명시, N11 피벗 alt.
- **C. 진짜 에이전트 오류 (2건)** — ①J02 **팬아웃 조인**: 카드×계좌 다대다 경유로 중복 집계
  → 평균 왜곡(100.2M vs 101.2M). 고전적 NL2SQL 실패 모드의 재현 — 이번 검증의 가장 가치 있는
  수확. ②N19: Description에 "(YYYYMM)" 명시돼 있는데 'YYYY-MM' 가정 → 0건. 조회는 했으나
  증거를 안 읽음. (가정 명시는 규율상 옳았음 — 증거 활용 실패로 분류)

### 교훈
- **1차 실측은 에이전트가 아니라 하니스를 측정한다** — oracle 자가검증으로도 못 잡는 것
  (의미 동치의 범위)은 실제 모델 출력이 와야 드러난다. 보정-재실행 사이클은 설계에 포함돼야 한다.
- 백분율 동치는 비율형(절대값<1 존재)에만 적용 — 건수 3 vs 300 오인을 차단하는 가드가 필수였다.
- J02 팬아웃은 프롬프트 패치로 가리지 않고 **역량 측정 항목으로 보존**한다 — 다대다 경유 시
  grain 검토는 에이전트가 스스로 해야 할 판단이며, 이를 프롬프트에 박으면 검증이 약해진다.
  (논의 필요 시 사용자 결정 사항으로 이관)

### 재실행 프로토콜
패치된 앱으로 N12·N25·G03만 단건 재실행 → 예상 베이스라인 ~41/44 (93%),
잔여 오답 J02·N19가 에이전트 역량 분석의 본 대상.

## 회고 6 — 데이터 탐색 v2 (다중 뷰)

### 검토 (v1의 문제)
테이블 단일 진입 — 개념에서 출발 불가, Term→자산 방향 비가시, 메트릭 11개 완전 비노출,
질문셋 비노출, 검색 부재, 코퍼스의 핵심 설계(동의어 충돌 구조)가 어떤 화면에도 없음.

### 설계 (과제 → 뷰)
과제 3개: 저자의 저작 감사 / 시연 관객의 세계 이해 / 오답 분석자의 트레이스→맥락 점프.
→ 뷰 5개(테이블·Term·메트릭·충돌 지도·질문셋) + 전역 검색 + 전 화면 교차 링크 + 해시 딥링크.

### 이터레이션 기록
- v2 초안 → 자기 비판: 서브탭 5개 과다 우려 → 멘탈 모델이 달라 통합이 더 혼란, 검색을
  만능 진입점으로 보완. 딥링크(#x=view/sel) 추가 — 오답 분석 공유용, 키 해시(#k=)와 패턴 분리.
- 잘라낸 것(명시): 메트릭 라이브 계산(실행식 미보유), 질문→Term 자동 매핑(데이터에 없음),
  모바일 레이아웃.
- 원칙 유지: 탐색 탭은 '레이어가 아는 만큼 + DB 원시 값'만. 결손 컬럼은 라벨 '?'로 자연 노출.
  family는 사람용 감사 정보로 표시하되 에이전트 연산(get_term)에서는 계속 비노출(D10 불변).

### 검증 (2단)

## 회고 8 — 대시보드 탭 + '중규모' 제거

### 동기
'중규모'라는 말이 기준점 없이 애매했다. 말로 뭉뚱그리는 대신 숫자로 보여주기로 하고,
데이터 탐색 첫 화면에 대시보드 탭을 신설.

### 설계 판단
- 위치: 데이터 탐색의 첫 번째 탭(기본 진입). 전체 그림 → 세부(테이블·Term) 흐름.
- 4개 섹션: 규모 / 레이어 커버리지 / 질문셋 / 설계된 모호성.
- 모든 수치는 런타임 계산 (하드코딩 0). v2 질문 수·분포가 자동 반영.

### 정직한 수치 (커버리지의 함정)
- Link 커버리지를 전체 컬럼 기준으로 재면 40%로 낮아 보임 — '충분히 채워진' 전제와 상충.
- 원인: PK/FK/감사 컬럼 149개(전체의 절반)는 Term을 안 거침.
- 해결: 비즈니스 컬럼(메타 제외) 기준 68%로 표시 + "PK/FK/감사 N개 제외" 명시.
  숫자를 유리하게 고른 게 아니라, 의미 있는 분모를 쓴 것. 전체 기준도 병기 가능하나
  비즈니스 기준이 '레이어가 의미 컬럼을 얼마나 덮는가'라는 질문에 정확.
